N2 migration plan: carry rotary switch position through InputSnapshot and WiFi scenarios

Goal
- Make rotary switch position a first-class input in InputSnapshot.
- Feed it from both live hardware and WiFi scenario profiles.
- Remove the WiFi-only display-selection hard-code so both builds use the same snapshot-driven path.

Feature branch
1. Create the branch from updated main:

   git checkout main
   git pull --ff-only
   git checkout -b feature/input-snapshot-rotary-switch

Scope of change
- src/InputSnapshot.h
- src/SystemProfile_minima.cpp
- src/SystemProfile_wifi_scenario.cpp
- N2V5.ino
- host_tests/test_BB_system_snapshot.cpp

Detailed plan

1) Add rotarySwitchStatus to InputSnapshot
- Edit src/InputSnapshot.h.
- Add a new field:
    uint8_t rotarySwitchStatus;
- Place it near blackSwitchEnabled so operator-controlled inputs stay grouped.

Expected result
- InputSnapshot can carry the currently selected rotary-switch/display position for both hardware and simulated modes.

2) Plumb live rotary value into InputSnapshot for Minima
- Keep readRotarySwitch() in N2V5.ino as the hardware-facing acquisition helper.
- In the live loop path, continue to read the rotary switch and store the result in:
    systemContext.runtime.display.rotarySwitchStatus
- Update src/SystemProfile_minima.cpp so systemProfileRefreshInputs(SystemContext&) copies:
    ctx.runtime.display.rotarySwitchStatus
  into:
    ctx.input.rotarySwitchStatus

Expected result
- The live profile publishes the rotary-switch position through the same input snapshot path used by the rest of the system.

3) Extend WiFi scenario data model
- Edit src/SystemProfile_wifi_scenario.cpp.
- Add rotarySwitchStatus to ScenarioStep.
- Update every ScenarioStep initializer in kScenario0 and kScenario1 to include a rotary value.
- Update applyCurrentStep() to assign:
    ctx.input.rotarySwitchStatus = step.rotarySwitchStatus;
- Update printScenarioLine() to print the simulated rotary value for traceability.

Expected result
- WiFi scenario mode can simulate turning the rotary switch to different display positions.

4) Remove WiFi-only display-selection hard-code
- Edit N2V5.ino.
- In displaySelectedValue(), stop hard-coding WiFi builds to config.display.rotaryN2Percent.
- Delete the board-specific logic that does this:
    #if defined(ARDUINO_UNOWIFIR4)
    systemContext.runtime.display.rotarySwitchStatus = systemContext.config.display.rotaryN2Percent;
    #else
    systemContext.runtime.display.rotarySwitchStatus = readRotarySwitch();
    #endif
- Change displaySelectedValue() to switch on snapshot-driven rotary position, preferably:
    systemContext.snapshot.input.rotarySwitchStatus
  or, if necessary:
    systemContext.input.rotarySwitchStatus

Expected result
- Both Minima and WiFi builds use the same selection logic.
- WiFi display mode is driven by scenario input, not by a compile-time hard-code.

5) Keep SystemRuntime display fields for now
- Do not remove these yet:
    runtime.display.rotarySwitchStatus
    runtime.display.previousButtonValue
- Continue to use them as:
  - acquisition/cache for live hardware input
  - change-detection/logging state inside readRotarySwitch()

Expected result
- Minimal safe refactor.
- No need to untangle live input acquisition and button change logging in the same step.

6) Update host-side regression coverage
- Edit host_tests/test_BB_system_snapshot.cpp.
- Extend the test data to set input.rotarySwitchStatus.
- Add assertions proving the value is preserved in the refreshed snapshot.

Expected result
- The new field is covered by at least one existing BB-style snapshot aggregation test.

Suggested initial scenario values
- For better manual verification, make the scenario steps exercise multiple display modes instead of always using the same one.
- Example progression:
    step 0: rotaryOff
    step 1: rotarySupply
    step 2: rotaryLeft
    step 3: rotaryRight
    step 4: rotaryN2Low
    later steps: rotaryN2Percent
- Use the existing config/display constants already defined by the project.

Implementation order
1. Add field to InputSnapshot.
2. Update BB snapshot test.
3. Extend ScenarioStep and scenario tables.
4. Update applyCurrentStep() and scenario logging.
5. Update SystemProfile_minima.cpp to copy live rotary value into ctx.input.
6. Remove WiFi hard-code from displaySelectedValue() and make it consume snapshot/input rotary state.

Verification checklist
1. Host tests pass:
   - host_tests/test_BB_system_snapshot.cpp updated and green.
2. Arduino Minima build compiles.
3. Arduino WiFi build compiles.
4. On Minima hardware, rotary selection still changes the displayed value.
5. In WiFi scenario mode, displayed value changes as the scenario rotarySwitchStatus changes.
6. Serial output in scenario mode shows the rotary value for each step.

Likely breakage points
- Any aggregate initialization of InputSnapshot must be updated for the new field.
- Every ScenarioStep initializer row must be updated to include the added field.
- Any tests that compare full InputSnapshot contents may need adjustment.

Suggested commit sequence
1. Add rotarySwitchStatus to InputSnapshot and snapshot test
2. Plumb rotarySwitchStatus through Minima and WiFi profiles
3. Make display selection snapshot-driven on all builds

End state
- Rotary switch position is treated like other external inputs.
- WiFi scenario mode can simulate the display-selector knob.
- Display logic is unified across hardware and simulated profiles.
