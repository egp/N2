N2 migration plan: remove direct Wire dependency from TM1650 path, verify hardware, then complete rotary-switch snapshot/scenario migration

Goal
- Make the N2 sketch talk to the TM1650 only through TCP1650 over TCP1819.
- Eliminate direct Wire use from the sketch’s 4-digit LED / rotary-switch path.
- First get the real TM1650 display + rotary path working on hardware.
- Then finish the InputSnapshot / WiFi scenario rotary-switch migration.

Current verified situation
- The sketch still directly depends on Wire for the TM1650 path.
- TCP1650 itself is already the correct transport abstraction and already uses TCP1819 only.
- InputSnapshot has already been locally updated to include rotarySwitchStatus.
- The current sketch likely has a rotary-code mismatch bug because it masks button reads with 0x3F while the configured rotary values are 0x44, 0x4C, 0x54, 0x5C, 0x64, 0x6C.

Branching
- Because you already changed InputSnapshot locally, branch from your current working tree so that change stays with the work.

Phase 1: remove direct Wire use from the TM1650 path

1) Stop including Wire in the sketch TM1650 path
- Edit N2V5.ino.
- Remove:
    #include <Wire.h>

Reason
- The sketch should not use Wire for the TM1650 if TCP1650/TCP1819 is the intended abstraction.

2) Stop initializing Wire in setup()
- Edit N2V5.ino.
- Remove:
    Wire.begin();
- Also remove the nearby commented Wire clock line rather than leaving stale transport diagnostics behind.

Keep
- setupI2C()
- I2CInit(&i2c_disp4, 100000)
- disp4.begin()

Reason
- setupI2C() already initializes the BBI2C bus used by TCP1650, so Wire.begin() is not part of the intended TM1650 path.

3) Replace readRotarySwitch() so it uses only TCP1650
- Replace the whole body of readRotarySwitch().
- New behavior:
  - call disp4.getButtons()
  - do not use Wire.requestFrom()
  - do not use Wire.read()
  - do not mask the result with 0x3F
  - keep the existing change-detection logging against previousButtonValue

Important rule
- Preserve the raw returned value from TCP1650::getButtons().
- Do not strip bit 6.
- Your configured selector values already include bit 6 set for pressed keys.

Expected result
- Rotary values can match the configured display selector constants again.

4) Do not change the TCP1650 library unless your installed local copy differs from the repo
- First verify your installed TCP1650 copy is the current TCP1819-only version.
- If your local installed library still references Wire, fix/sync that library first.
- If your local installed library matches the current repo, then N2 should be fixed only by changing the sketch.

Suggested local verification
  rg -n "Wire" ~/Documents/Arduino/libraries/TCP1650
  rg -n "I2CWrite|I2CRead|getButtons" ~/Documents/Arduino/libraries/TCP1650/src

Phase 2: verify hardware path before touching scenarios further

5) Verify the TM1650 display path still works with TCP1650 only
- Build and run on Minima hardware.
- Confirm setup() still does:
  - disp4.begin()
  - enableDisplay4()
  - displaySelfTest()
- Confirm the 4-digit display shows the self-test and normal updates.

6) Verify rotary codes on real hardware
- Use the serial logging already present in readRotarySwitch().
- Turn the selector through all positions.
- Confirm the raw values now line up with the configured constants:
  - rotaryOff        0x44
  - rotarySupply     0x4C
  - rotaryLeft       0x54
  - rotaryRight      0x5C
  - rotaryN2Low      0x64
  - rotaryN2Percent  0x6C

Decision rule
- If the raw values match those constants, keep the config values unchanged.
- If they do not match, update the config constants in the profile config, not the transport layer.
- Do not reintroduce masking.

Phase 3: finish the live-input rotary migration

7) Move rotary acquisition out of displaySelectedValue()
- displaySelectedValue() should become render-only.
- It should not poll hardware.
- The real hardware rotary should be acquired earlier in the loop, before snapshot refresh.

Implementation direction
- In the Minima/non-WiFi loop path:
  - readBlackSwitch()
  - readPressureSensors()
  - read the rotary switch once and store it in:
      systemContext.runtime.display.rotarySwitchStatus
  - then call systemProfileRefreshInputs(systemContext, timerClock)

Expected result
- Rotary position becomes part of the same sampled input state as the sensor readings.

8) Update src/SystemProfile_minima.cpp to copy the live rotary value into InputSnapshot
- In systemProfileRefreshInputs(SystemContext&, ProfileClock&):
  - keep sampledAtMs and all existing pressure copies
  - add:
      ctx.input.rotarySwitchStatus = ctx.runtime.display.rotarySwitchStatus;

Expected result
- InputSnapshot becomes the authoritative source for the current rotary-switch position in live mode.

9) Change displaySelectedValue() to consume InputSnapshot
- Remove the existing board-specific acquisition logic from displaySelectedValue().
- Delete the WiFi hard-code that forces rotaryN2Percent.
- Delete the direct call to readRotarySwitch() from displaySelectedValue().
- Switch on:
    systemContext.snapshot.input.rotarySwitchStatus
  instead of:
    systemContext.runtime.display.rotarySwitchStatus

Expected result
- The display logic becomes profile-agnostic and snapshot-driven on both Minima and WiFi.

Phase 4: complete WiFi scenario support

10) Extend ScenarioStep with rotarySwitchStatus
- Edit src/SystemProfile_wifi_scenario.cpp.
- Add:
    uint8_t rotarySwitchStatus;
  to struct ScenarioStep.

11) Update every scenario row
- Add rotarySwitchStatus to each initializer in kScenario0 and kScenario1.
- Use a deliberate sequence that exercises the selector:
  - off
  - supply
  - left
  - right
  - low N2
  - N2 percent

12) Plumb scenario rotary values into InputSnapshot
- In applyCurrentStep(SystemContext&, ProfileClock&):
  - after blackSwitchEnabled
  - assign:
      ctx.input.rotarySwitchStatus = step.rotarySwitchStatus;

13) Print the simulated rotary value in scenario logs
- Update printScenarioLine(const SystemContext&) to print:
    ctx.input.rotarySwitchStatus

Expected result
- WiFi scenario mode can simulate the rotary selector, and serial logs will show which mode each scenario step is using.

Phase 5: regression coverage

- Add:

Verification checklist

A. Hardware-path cleanup
- N2 no longer includes <Wire.h>.

- readRotarySwitch() uses disp4.getButtons() only.

- Minima copies runtime.display.rotarySwitchStatus into input.rotarySwitchStatus.

- Snapshot test is green.

3. Move live rotary acquisition into sampled inputs
4. Add rotarySwitchStatus to WiFi scenario steps and logs
5. Extend BB snapshot test for rotarySwitchStatus

End state
- The sketch talks to the TM1650 only through TCP1650 over TCP1819.
- The rotary selector works on hardware using raw TM1650 key codes.
- InputSnapshot carries rotarySwitchStatus for both live and scenario builds.
- Display selection is unified and snapshot-driven.
